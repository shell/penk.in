Rake Interactive

by Vladimir Penkin - Sep 9th, 2012 -

Motivation

Motivation behind this task is something that bothers me for a long time.

I bet you running in this situation fairly often. I even find this in lines in installation instructions for ruby(rails) projects.

How many times rails environment is getting loaded? Let’s count!

4 times we are loading rails environment. But do we really need to load it all the time? what if we forgot to do ‘rake stats’?

Rails environment loads up for 11 seconds. 11 * 5 = 55. It is minute of waiting and staring at screen.

Can we do better? Of course!

Rake interactive

Here is simple rake task that waiting for commands to execute.

Of course this is very simple script, there is no support of history or chain execution. Can we do better?

This version accepts optional parameter - string of commands separated by &&

Good eh? 11 seconds for rails to start and 2 more seconds for execute database tasks. Instead of 55 seconds.

Is it the end?

There are few questions left with this script. Model or Tasks caching - might be we need to implement something like reload! function in interactive console to reload rails models.

But the bigger question is that you might don’t want to run yet another rails instance when you already running rails server and rails console. There seems no problem with running Rake::Task[‘TASKNAME’].invoke from console. But you know what would be really cool? Run that from rails server!

I am not sure if this can be implemented. gets is blocking operation so while waiting for user input, we will block whole server. It might be possible with experimental rails projects like async-rails or servers like unicorn or thin.

So if you have some idea how to do that, just let us know!
